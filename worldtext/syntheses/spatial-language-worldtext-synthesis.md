## What the Table Proves

If the morphism holds — if every program preserves its invariant and failure mode across LDraw, Worldtext, and LEGOS — then these are not three different tools. They are three **views** of one canonical program.

But not all 21 rows hold equally. Three mappings are **forced, weak, or only partial**. We show them because a morphism table that never admits failure is a brochure, not a proof.

### Where the Morphism Breaks

**Program 14 (Spatial Hypertext) → LDraw: WEAK.**

Spatial hypertext allows ambiguity — you place items loosely, clusters emerge before formalization. LDraw does *not* allow this. Every part reference in an LDraw file has an exact coordinate. There is no "loose placement" in LDraw; there is only placement or absence. The mapping works for Worldtext (where LEGOS blocks can be provisionally placed before formal wiring) and for context engineering (where prompt blocks can be loosely grouped before structuring). But for LDraw specifically, the program doesn't map. LDraw skips the ambiguity stage entirely. The invariant ("structure may emerge before it is demanded") is violated — LDraw demands structure from the first line.

**Program 15 (Image-Schema) → LDraw: PARTIAL.**

Image-schemas are embodied spatial primitives (CONTAINER, SOURCE-PATH-GOAL, BLOCKAGE). LDraw has containers (submodels) and paths (step sequences), but it has no concept of BLOCKAGE — nothing in the LDraw format represents "this part *prevents* movement through this space." The constraint must be inferred by the operator, not parsed by the format. In Worldtext, BLOCKAGE maps cleanly to rules (`INVARIANT: No agent may touch the tide`). In LDraw, the closest equivalent is a physical collision between bricks — but that's geometry, not grammar. The mapping is forced.

**Program 13 (Gothic Assemblage) → Worldtext: UNCERTAIN.**

Gothic assemblage says: "large structure emerges from local actors obeying shared grammar." LDraw proves this — thousands of part authors, one file format, a shared library that grew over 30 years. But does *Worldtext* actually work this way? The worked examples in [[syntheses/hello-worldtext]] are all single-operator demonstrations. No multi-author Worldtext has been stress-tested. We claim the program transfers. We have not yet proven it does. The mapping is a prediction, not an observation.

*(An earlier version of this table had all 21 rows marked as clean mappings. Reviewing the table against the audit criteria in [[syntheses/worldtext-red-team]], we realized that the Spatial Hypertext and Image-Schema rows were forced fits — the LDraw format does not support ambiguity or blockage natively. This was initially uncomfortable, but it clarified something: LDraw is a Program 2/4/5/7 system (coordinate, reference, recursion, canonical artifact). It was never designed to handle Programs 14 or 15. The fact that Worldtext handles them and LDraw does not is precisely what makes Worldtext necessary — it extends the grammar where LDraw stops.)*

The canonical program is:

> **Spatial language turns thought into placed, referenced, inspectable, governed, repairable structure.**

The full 24-program theory is documented in [[syntheses/spatial-language-program-theory]].

LDraw is the proof that this works for physical objects.  
Worldtext is the proof that this works for narrative worlds.  
LEGOS is the proof that this works for any concept.

## The Dual Compilation Loop (Revisited)

The LDraw/diffusion experiment from Chapter 01 is now explicable:

```
<same text>
  ├── [compiles via] LDraw → deterministic 3D render (canonical artifact, program 7)
  └── [generates via] diffusion → stochastic image (regenerable view, program 7)

<deterministic render> [constrains] <stochastic generation> (program 20: worlding)
<stochastic generation> [reveals] <implications of deterministic render> (program 10: professional vision)
<operator> [inspects both] → [repairs source] → [regenerates views] (program 21: repairable world)
```

The loop works because both pipelines share the same canonical artifact (the text). The compile pipeline enforces the **coordinate, reference, and ownership** programs (2, 4, 8). The generation pipeline tests the **professional vision, breakdown-visibility, and parse-don't-validate** programs (10, 11, 9). The operator's repair closes the circuit through the **provenance** program (12).

This is why the two images from the `LDRAW WORLD` directory are not "different versions of the same thing." They are **different views of the same canonical source**, each revealing what the other hides:

| | LDraw Compile | Diffusion Generation |
|---|---|---|
| **What it shows** | Exact positions, part IDs, rotation matrices, color codes | Surface textures, lighting, material properties, atmospheric depth |
| **What it hides** | Material qualities, environmental context, implied aesthetics | Exact coordinates, part provenance, ownership graph |
| **Which programs it tests** | 1 (inscription), 2 (coordinate), 4 (reference), 5 (recursion), 6 (local dictionary) | 10 (professional vision), 11 (breakdown-visibility), 20 (worlding) |
| **What failure looks like** | Missing part file, clipping geometry, null reference | Hallucinated entity, broken spatial constraint, generic slop |

### Rules (from spatial constraints)

```

  description: "Viewers are OUTSIDE the glass. The ecology is INSIDE."
  if: <entity>.type == viewer
  then: <entity>.z > 200 OR <entity>.z < -200
        // wall is at z = ±200; viewers are at z = ±260
  source: "0 OUTSIDE VIEWERS — ALL OUTSIDE the glass, facing IN toward it"
}

rule <Terminal_Orientation> {
  description: "Terminals face OUTWARD so viewers can see them through glass."
  if: <entity>.type == terminal AND <entity>.location == <Glass_Conservatory>
  then: <entity>.rotation == "facing nearest wall"
  source: "0 INSIDE WALL: TERMINALS POINTED OUTWARD"
}

rule <Containment> {
  description: "The baseplate is the world boundary. Nothing exists beyond it."
  if: <entity>.position exceeds <Paradisa_Baseplate>
  then: event <Nullify_Vector>
  // analogous to Oceanus on the Shield
}
```

### Relations

```
rel [views_through](<Shark_Minifig> -> <Computer_Flowers>)   // viewer looks in
rel [faces](<Terminals> -> <Glass_Walls>)                     // terminals face out
rel [surrounds](<Forest_Ring> -> <Glass_Conservatory>)        // trees ring the glass
rel [contains](<Glass_Conservatory> -> <Inner_Garden>)        // glass contains garden
rel [marks](<Heraldic_Shields> -> <Glass_Walls>)              // shields on the boundary
```

### The Proof

Change `rule <Terminal_Orientation>` — make the terminals face INWARD instead of outward.

**Propagation:**
- The viewers can no longer see the screens → the "looking in" relation breaks
- The heraldic shields on the glass become the only outward-facing information
- The conservatory becomes opaque from outside (you see blank backs of monitors)
- The function shifts from "exhibition" to "exclusion" — the glass becomes a barrier, not a window

One rotation matrix change. One rule mutation. The entire meaning of the spatial arrangement cascades. **The Worldtext is alive.**

## The Core Discovery

The spatial language program theory and the Worldtext framework are the same theory at different altitudes.

**Spatial language** says: make thought behave like architecture.  
**Worldtext** says: make text behave like a world.  
**LDraw** says: make description behave like a build instruction.

They converge on one law:

> **Do not merely describe the world. Keep the language that can regenerate it.**

The LDraw `.mpd` file can regenerate the 3D model.  
The Worldtext can regenerate the constrained narrative.  
The LEGOS grammar can regenerate the conceptual structure.

If the language is lost, only the surface remains. And surfaces cannot be repaired. They can only be replaced.

This is why the "Hello, Worldtext" test matters. It is the proof that the regeneration language is intact — that the text is not wallpaper but source code, not decoration but constitutional law, not a painting of a world but the grammar that compiles one.
